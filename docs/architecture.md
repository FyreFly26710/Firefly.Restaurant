# Architecture

This document records the Firefly Restaurant architecture that agents and developers should preserve.

## System Overview

Firefly Restaurant has two initial project areas:

- `src/client/web`: public restaurant storefront.
- `src/server`: Blazor admin pages and .NET APIs in one solution.

The public storefront is static-first.
Visitors should receive HTML, assets, and media from Cloudflare whenever possible.
The .NET server owns admin workflows, persistence, media processing, API boundaries, and revalidation triggers.

`docs/demo-ui` is a prototype reference for UX direction only.
Its sample restaurant identity, names, and copy are not canonical Firefly product data.

## Frontend

The storefront uses Next.js App Router with React, TypeScript, and Tailwind CSS.

Core frontend rules:

- Keep page and layout modules thin.
- Put feature behavior under `src/features`.
- Keep server-side data access in dedicated data/API modules.
- Prefer static rendering, ISR, and cache tags/paths for public content.
- Do not make browser-side storefront code call the .NET API for normal browsing.
- Use client components only where interactivity requires them.
- Use Playwright for route smoke tests and critical journeys.

Cloudflare deployment is Pages-first for now.
This is an explicit project choice with a validation risk: current Cloudflare guidance directs full-stack Next.js ISR support toward Workers/OpenNext, while Pages is best suited to static output.

## Backend

The server area uses one .NET 10 `.slnx` under `src/server`.

Planned projects:

- `gateway.api`: public or edge-facing API gateway when API exposure is needed.
- `admin.app`: Blazor Server admin UI.
- `menu.api`: menu/content API boundary.
- `menu.core`: menu/content domain and application logic.
- `user.api`: user/account API boundary.
- `user.core`: user/account domain and application logic.
- `common.lib`: shared kernel for small stable cross-cutting code.

Backend rules:

- Keep `Program.cs` files thin.
- Use MediatR for write commands.
- Use explicit query services for reads.
- Keep EF Core persistence in infrastructure boundaries.
- Use Problem Details for API errors.
- Use OpenAPI for HTTP APIs.
- Keep auth, correlation, logging, validation, and exception handling centralized.
- Avoid splitting runtime deployment just because projects are separated in the solution.

## Data

PostgreSQL is the default relational store.
EF Core owns persistence and migrations.

Menu and shop content are the first public content domains.
The server owns canonical content, media metadata, and admin permissions.
The storefront consumes build-time, server-side, or revalidation-safe read models.

Shared TypeScript contracts may be introduced later when they are useful for web or future mobile clients, but C# backend contracts remain the server source of truth.

## Operations

The local server should be reachable through Cloudflare Tunnel, not directly exposed through router ports.

Cloudflare should cache:

- storefront pages and assets,
- CDN-safe media routes,
- static files generated by the web build.

Admin actions that affect public content should trigger storefront revalidation.
Timed revalidation remains the fallback path.

Public `/api/*` exposure is not a default.
If an issue adds public runtime APIs, it must document cache behavior, rate limiting, authentication needs, and abuse protections.

## Architecture Decisions

- Keep the repository layout under `src/`, not an `apps/` monorepo.
- Keep root skills under `.agents/skills`.
- Keep project-specific docs under `docs/client/web` and `docs/server`.
- Start server work as one .NET solution with multiple projects.
- Treat Cloudflare Pages as the initial web deployment target, while tracking Workers/OpenNext as a likely ISR validation path.
